## 2. 架构审阅

### 2.1 确认可用（无需调整）

以下部分经审阅确认合理，可直接进入实现：

- **分层架构**：React 前端 / Tauri 核心 / Rust 后端，职责清晰
- **多窗口策略**：`island`（主控件）+ `pin-<id>`（贴图）+ `snip`（截图选区）+ `launcher`（P2 搜索），每个窗口独立设置属性，互不干扰
- **技术栈选型**：Tauri v2 / Rust / React 18 / TypeScript / Zustand / framer-motion / rusqlite(bundled) / arboard / windows crate — 无明显问题
- **分阶段路线图**：P0（MVP）→ P1（微件+截图OCR）→ P2（录屏+搜索）→ P3（多屏+迁移+优化）— 节奏合理
- **数据模型**：`config` 键值表 + `pins` 贴图表 — 满足 P0 需求
- **前后端通信**：invoke（前端主动请求）+ emit/listen（后端主动通知）— 规范

### 2.2 需细化（5 项）

以下部分方向正确但细节不足，需在 P0 spec 中补充：

| # | 问题 | 影响 | 细化建议 |
|---|------|------|----------|
| A1 | **配置跨窗口传播机制缺失** — Zustand 是单窗口状态，灵动岛主窗口与 N 个贴图窗口之间如何同步配置变更？ | 用户在设置中改变配置后，已打开的贴图窗口不会自动更新 | Rust 端在配置变更时 `emit("config-changed", { key, value })`，各窗口 listen 后按需刷新本地 Zustand 状态。或：各窗口不缓存配置，按需从 DB 读取（适合低频配置） |
| A2 | **形变动画与 `setSize` 同步策略不够明确** — framer-motion `layout` 动画在 CSS 层面过渡，但 Tauri 窗口尺寸需手动 `setSize`。若 `setSize` 滞后于 CSS 动画，内容被裁切；若提前，出现空白区域 | 灵动岛形态切换时视觉跳跃/裁切 | 用 `ResizeObserver` 监听内容实际尺寸变化，在动画关键帧同步 `setSize`；或用 framer-motion `onUpdate` 回调驱动窗口尺寸。原文档"动画结束回调校正"可能产生跳跃，不推荐 |
| A3 | **剪贴板去重策略未具体化** — 文档提"感知哈希或尺寸+部分像素"但未确定 | 重复图片可能多次触发贴图提示 | MVP 用「图片尺寸（宽×高字节）+ 前 1024 字节 hash」快速去重。感知哈希（pHash）留 P1+ 优化。去重在 Rust 端完成 |
| A4 | **贴图 z-order 管理未明确** — 文档说"各自独立 z-order"，但 Tauri `setAlwaysOnTop` 是布尔值非层级 | 用户可能期望调整贴图之间的前后关系 | MVP 只做「置顶/不置顶」二态切换，不做 pin 间 z-order 调整。点击某个 pin 时 `setFocus` 使其置于同层最前。P3 评估是否需要层级面板 |
| A5 | **单实例二次启动行为未定义** — 岛是 `skipTaskbar`，`setFocus` 可能不符合预期 | 二次启动时用户无感知 | 二次启动时 emit 事件让岛做"闪动"动画提示用户（如缩放脉冲一次），而非直接 `setFocus` |

### 2.3 缺失（4 项）

以下部分在原文档中完全未覆盖，需补充：

| # | 缺口 | 影响 | 补充建议 |
|---|------|------|----------|
| A6 | **测试策略** | 无法保证代码质量，重构困难 | **Rust 端**：`#[cfg(test)]` 单元测试，覆盖 DB 操作（建表/迁移/CRUD）、图片处理（DIB→PNG 转换）、去重逻辑（hash 对比）。**前端**：Vitest + React Testing Library，覆盖组件交互（形态切换、工具条操作）。**Tauri 命令层**：集成测试验证 invoke/event 流程。MVP 至少覆盖 DB 层 + 去重逻辑 |
| A7 | **错误处理与日志** | 出问题时无法定位；错误体验差 | **Rust 端**：`tracing` crate 结构化日志（输出到 `AppData/logs/app.log`，按日轮转）+ `thiserror` 定义业务错误枚举。**错误传播**：Rust 命令返回 `Result<T, AppError>`，前端 invoke catch 后 Toast 提示用户。**前端**：`console.error` + 全局 ErrorBoundary |
| A8 | **自动更新** | 用户需手动下载新版本 | P0 不做。预留 `tauri-plugin-updater` 接口位（配置中留 `update_url` 字段）。P3 实现。签名验证与版本管理在 P3 设计时细化 |
| A9 | **i18n** | 未来多语言支持困难 | MVP 纯中文，不引入 i18n 框架。但前端文案集中管理到 `src/locales/zh.ts`（key-value 形式），为未来留接口。P3 评估是否引入 `i18next` |

### 2.4 审阅结论

原始文档整体质量高，架构方向正确。9 项需细化/补充的点中：

- **A1/A2/A6/A7** 需在 Phase 2 完整 spec 中明确写入
- **A3/A4/A5** 需补充到现有子系统方案中
- **A8/A9** 仅需预留接口位，P0 不实现
